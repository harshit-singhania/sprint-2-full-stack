## 1. Project Overview

**What it does:** A backend REST API for a used-car buy-and-sell marketplace.

**Three user roles:**
- `ADMIN` — approves/rejects car listings and purchase orders, views all dashboards, manages users, manages support tickets
- `USER (Seller)` — lists cars, approves/rejects sale requests on their own listings
- `USER (Buyer)` — browses approved cars, purchases, manages wishlist, raises support tickets

> A single `USER` account can act as both seller and buyer simultaneously.

> Note: Both Seller and Buyer share the `USER` role. A single user can list cars AND buy cars — the app distinguishes them by context (who owns the car in a given transaction).

**Architecture:** Single-module Spring Boot monolith. No JWT — uses a custom stateful session-token system. No Spring Security filter chain — auth is enforced manually in the service layer.

## 5. DTOs

DTOs are Java **records** (immutable, auto-generated equals/hashCode/toString).

### `RegisterRequest`
```
username (NotBlank), name (Title Case pattern), phoneNumber (Indian 10-digit regex),
email (@Email, optional), password (NotBlank), role (NotBlank)
```
- Phone regex: `^[6-9]\d{9}$` — Indian mobile numbers start with 6-9
- Name regex: `^[A-Z][a-z]+(?: [A-Z][a-z]+)*$` — enforces Title Case

### `LoginRequest`
```
username (NotBlank), password (NotBlank)
```

### `LoginResponse`
```
message, sessionToken (raw token — sent once, never stored raw), role
```

### `CarRequest`
```
make, model, year (1900–2100), price (min 0.01), mileage (min 0), color, available (nullable)
```

### `PurchaseRequest`
```
paymentMethod (NotBlank), paymentToken (optional), paymentSuccessful (optional Boolean)
```
Two modes: modern `paymentToken`-based, or legacy `paymentSuccessful` boolean flag.

### `AdminDashboardResponse` / `SellerDashboardResponse`
Aggregated stats records — see fields in the model section.

### `ApiMessage`
Single-field record: `message`. Used as a generic success/error wrapper.

## 9. Service Layer

### `SessionTokenService`
A `@Service` bean (not behind an interface) that handles raw token operations.

- **`generateToken()`** — generates 32 cryptographically random bytes, URL-safe Base64 encoded. Result is a 43-char string.
- **`hashToken(token)`** — SHA-256 hash of the token, returned as a lowercase hex string. This is what's stored in the DB.

**Why not JWT?** This app uses stateful sessions — the token itself is meaningless; only its hash in the DB gives it meaning. This allows forced logout (clear the hash from DB) which JWT can't do without a blocklist.

---

### `SessionServiceImpl` (Authentication Guard)

Every protected service method calls one of three guard methods first:

| Method | What it checks |
|---|---|
| `requireUser(token)` | Valid token + not expired → returns `AppUser` |
| `requireRole(token, role)` | Same + checks exact role match |
| `requireAnyRole(token, roles...)` | Same + checks against a list of allowed roles |

**Expiry logic:** If `activeSessionExpiresAt` is in the past, the session is cleared from DB immediately and `401` is thrown. Session expiry is a soft delete — the user can log in again.

---

### `AuthServiceImpl`

#### `register(RegisterRequest)`
1. Check `username` uniqueness → 409 if duplicate
2. Check `phoneNumber` uniqueness → 409 if duplicate
3. Check `email` uniqueness (only if provided) → 409 if duplicate
4. Parse role string → 400 if invalid
5. If role is `ADMIN`, check no admin exists yet → 409 (only one admin allowed)
6. Hash password with BCrypt
7. Save `AppUser`
8. Fire `notificationService.notifyUserRegistered()`
9. Return `ApiMessage("Registration successful...")`

#### `login(LoginRequest)`
1. Find user by username → 401 if not found
2. BCrypt `matches(rawPassword, hash)` → 401 if wrong
3. Generate new token via `SessionTokenService`
4. Store hash + expiry in user record
5. Return `LoginResponse(message, rawToken, role)` — the **only** time the raw token is sent

#### `logout(String token)`
1. Hash the incoming token, look up user by hash → 401 if not found
2. Null out `activeSessionTokenHash` and `activeSessionExpiresAt`
3. Save user

---

### `CarServiceImpl`

#### `addCar(token, CarRequest)`
- Requires `USER` role
- Sets `available = false`, `approvalStatus = PENDING_ADMIN_APPROVAL`
- Sets `seller = authenticated user`
- Saves and returns the car

#### `updateCar(token, carId, CarRequest)`
- Requires any logged-in user
- Non-admin users can only update their own cars (`ensureCarOwnerOrAdmin`)
- Non-admin update re-triggers approval: `available = false`, status = `PENDING_ADMIN_APPROVAL`
- Admin can update any car without re-triggering approval

#### `deleteCar(token, carId)`
- Owner or admin can delete

#### `browseAvailable(token)`
- Admins see all `APPROVED + available` cars
- Regular users see `APPROVED + available` cars **excluding their own listings** (can't buy what they're selling)

#### `viewDetails(token, carId)`
- Non-approved cars visible only to admin or the car's own seller
- Increments `viewCount`
- Creates a `RecentView` record (stores token **hash**, not raw token)

#### `recentViews(token)`
- Queries last 10 `RecentView` by the session's token hash

#### `compare(token, firstCarId, secondCarId)`
- Returns a list of two cars — UI handles the actual comparison display

#### `popular(token)`
- Returns top 5 cars by `viewCount` DESC

#### `pendingApproval(token)` / `approveListing(token, carId)` / `rejectListing(token, carId)`
- Admin only
- Approve sets `available = true`, `approvalStatus = APPROVED`
- Reject sets `available = false`, `approvalStatus = REJECTED`

---

### `PurchaseServiceImpl`

#### `purchase(token, carId, PurchaseRequest)` — the most complex flow

```
1.  requireRole(USER)                         ← must be USER, not ADMIN
2.  getCar(carId)                             ← 404 if not found
3.  buyer != seller check                     ← can't buy own car
4.  car.available && car.APPROVED check       ← 400 if not on market
5.  paymentGateway.process(...)               ← hit SimulatedPaymentGateway
6.  Create & save Payment record              ← always saved, SUCCESS or FAILED
7.  If payment FAILED → return ApiMessage     ← no order created
8.  car.available = false → save car          ← lock car immediately
9.  Create PurchaseOrder                      
10. fraudAlert = (buyer's total orders > 3)  ← auto fraud flag
11. order.status = PENDING_ADMIN_APPROVAL
12. Save order, link payment.orderRef
13. notifyPurchaseCreated()
14. Return saved order
```

#### Order approval flow (two-step)
- **Admin approves** → status: `PENDING_ADMIN_APPROVAL` → `PENDING_SELLER_APPROVAL`
- **Seller approves** → status: `PENDING_SELLER_APPROVAL` → `APPROVED`; car.available = false (confirmed sold)
- Either side can **reject** at their stage → status: `REJECTED`

#### Fraud Detection
Auto-flags orders where the buyer already has more than 3 total orders. Sets `fraudAlert = true` on the `PurchaseOrder`. Admin can see the count in the dashboard.

---

### `DashboardServiceImpl`

**Seller Dashboard** — requires USER role, aggregates stats for *that seller's* cars and orders:
- Total/available/sold/pending/rejected listings
- Order counts by status (pending admin, pending seller, approved, rejected)
- Fraud alert count
- Total revenue (sum of payment amounts on APPROVED orders)

**Admin Dashboard** — requires ADMIN role, platform-wide stats:
- User counts (total, admin, user)
- Car counts (total, available, sold, pending, rejected)
- Order counts by every status
- Fraud alert count
- Ticket counts (open, closed)
- Total platform revenue

---

### `FeedbackServiceImpl`

- `submit()` — any authenticated user can submit. No duplicate check.
- `all()` — admin only, returns all feedback records.

---

### `SupportTicketServiceImpl`

- `create()` — USER role only. Status starts as `OPEN`.
- `mine()` — USER role: returns only their own tickets.
- `all()` — ADMIN only: returns all tickets.
- `update()` — ADMIN only: can change status (OPEN/CLOSED) and/or add a response message.
- `addMessage()` — any authenticated user, but only the ticket's owner or an admin. Adds a `TicketResponse` to the ticket's collection (cascade saves automatically).

---

### `WishlistServiceImpl`

- `add()` — USER role only. Car must be `APPROVED` and `available`. Uses `orElseGet` — idempotent, no error if already in wishlist.
- `remove()` — USER role. Car must be in their wishlist → 404 if not.
- `list()` — USER role. Returns their entire wishlist.

---

### `NotificationService`

- Not behind an interface — direct `@Service` bean.
- `emailEnabled` flag — when `false`, logs what *would* be sent instead of actually sending.
- `mailSender` is `@Autowired(required = false)` — app starts even without SMTP config.
- Sends to the user on registration and to all ADMIN users when a new purchase order is created.
- Silently skips if recipient email is blank.

---

### `SimulatedPaymentGateway`

Implements `PaymentGateway` interface. In production you'd swap this for a real gateway bean.

**Decision logic:**
1. If `paymentToken` is present and starts with `"fail"` (case-insensitive) → **FAILED**
2. If `paymentToken` is present and doesn't start with `"fail"` → **SUCCESS**
3. If `legacyPaymentSuccessful == false` → **FAILED**
4. Otherwise → **SUCCESS**

Returns a `PaymentGatewayResult` record with `(successful, gatewayName, transactionId, message)`.

## 11. Complete Data Flow Diagrams

### Registration & Login Flow

```
Client                   AuthController        AuthServiceImpl         DB
  |                           |                      |                   |
  |-- POST /api/auth/register ------>                |                   |
  |   {username, name, phone,  |                     |                   |
  |    email, password, role}  |                     |                   |
  |                           |--- register(req) --->|                   |
  |                           |                      |-- existsByUsername? -->|
  |                           |                      |<-- false ----------|
  |                           |                      |-- existsByPhone? -->|
  |                           |                      |<-- false ----------|
  |                           |                      |-- existsByEmail? -->|
  |                           |                      |<-- false ----------|
  |                           |                      |-- countByRole(ADMIN)? (if admin) -->|
  |                           |                      |-- BCrypt.encode(password)          |
  |                           |                      |-- userRepository.save(AppUser) -->|
  |                           |                      |<-- savedUser ---------|
  |                           |                      |-- notifyUserRegistered() (email/log)
  |<-- 200 {"message":"Registration successful"} ----|                   |
  |                           |                      |                   |
  |-- POST /api/auth/login ------->                  |                   |
  |   {username, password}    |                      |                   |
  |                           |--- login(req) ------->|                   |
  |                           |                      |-- findByUsername ->|
  |                           |                      |<-- AppUser --------|
```

## 15. Interview Q&A

---

**Q1: What is the overall architecture of this application?**

A: It's a monolithic Spring Boot REST API. All features — auth, car management, purchases, payments, support, wishlist — live in a single deployable JAR. The layered architecture is: Controller → Service Interface → Service Implementation → Repository → Derby DB. There is no microservice split, no message queue, no external auth provider.

---

**Q2: How does authentication work? Why not use JWT or Spring Security?**

A: The app uses a custom stateful session token system. On login, a 32-byte cryptographically random token is generated using `SecureRandom`. The raw token is sent to the client once. On the server, only its SHA-256 hash is stored in the `users` table alongside an expiry timestamp. On every subsequent request, the client sends the raw token in the `X-Session-Token` header. The server hashes it and looks it up in the DB. This approach was chosen over JWT because it allows immediate forced logout — clearing the hash from the DB invalidates the session instantly, which pure stateless JWTs can't do without a blocklist.

---

**Q3: How is role-based access control enforced without Spring Security?**

A: Manually, in the service layer. The `SessionService` interface exposes three methods: `requireUser(token)` for any authenticated user, `requireRole(token, role)` for an exact role match, and `requireAnyRole(token, roles...)` for multiple allowed roles. Every protected service method calls the appropriate guard first. There are no filter chains or annotations — it's pure Java logic.

---

**Q4: Walk me through the purchase flow.**

A: First, the buyer's session is validated and must be USER role. The car must be approved, available, and not owned by the buyer. The payment is processed through the `PaymentGateway` (simulated). A `Payment` record is always saved — even on failure — for audit purposes. If payment fails, the flow stops and a message is returned. If payment succeeds, the car is immediately marked unavailable (to prevent race conditions with concurrent purchases), a `PurchaseOrder` is created with status `PENDING_ADMIN_APPROVAL`, and a fraud check is run. The order goes through a two-step approval: Admin first, then Seller. Only after both approve does the order reach `APPROVED` status.

---

**Q5: What is the fraud detection logic?**

A: It's a simple heuristic — if a buyer has placed more than 3 total orders at the time of a new purchase, the new order's `fraudAlert` flag is set to `true`. This is flagged on the order record and counted in the admin dashboard. It doesn't block the purchase but highlights suspicious patterns for manual admin review.

---

**Q6: How does the simulated payment gateway work?**

A: `SimulatedPaymentGateway` implements the `PaymentGateway` interface. It supports two modes: if a `paymentToken` string is provided and starts with "fail", the payment fails. Any other non-null `paymentToken` succeeds. As a legacy mode, a `paymentSuccessful` boolean can be passed directly. The interface abstraction means a real gateway implementation can be swapped in without changing any service code.

---

**Q7: Why is `@Enumerated(EnumType.STRING)` used instead of the default `ORDINAL`?**

A: `EnumType.ORDINAL` stores integers (0, 1, 2...). If you ever add a new value in the middle of an enum, existing DB rows become corrupt — their integer values now point to the wrong enum constant. `EnumType.STRING` stores the name (`"APPROVED"`, `"REJECTED"`), so adding new values is always safe.

---

**Q8: What is AOP and how is it used here?**

A: AOP (Aspect-Oriented Programming) allows cross-cutting concerns — like logging — to be applied uniformly without polluting business code. The `LoggingAspect` uses `@Around` advice with a pointcut that matches all controller and service methods. It measures execution time and logs it. One `@Around` handles both the success path (logs duration) and the failure path (logs the error and rethrows), avoiding the `@AfterThrowing` double-logging problem.

---

**Q9: How does the recent views feature track a session without storing the raw token?**

A: The `viewDetails()` method in `CarServiceImpl` stores `sessionTokenService.hashToken(token)` in the `RecentView.sessionToken` field — not the raw token. When `recentViews()` is called, it hashes the incoming token the same way and queries `findTop10BySessionTokenOrderByViewedAtDesc(hash)`. The two hashes match because SHA-256 is deterministic. This means even if the `recent_views` table is compromised, no active session tokens are exposed.

---

**Q10: Can a seller buy their own car?**

A: No. In `PurchaseServiceImpl.purchase()`, there's an explicit check: `car.getSeller().getId().equals(buyer.getId())` → throws 400 "User cannot buy a car they are selling." Also, `browseAvailable()` filters out the requesting user's own listings for USER role, so sellers don't even see their own cars in the browse list.

---

**Q11: What happens if two buyers try to purchase the same car simultaneously?**

A: The car is set to `available = false` immediately when the first purchase goes through (before the order is saved). The DB transaction ensures this is atomic. If a second buyer queries the car after that, they'll find it unavailable and get a 400 error. However, there's no explicit DB-level row-locking (`@Lock`), so in a high-concurrency scenario under the same transaction, a race condition is theoretically possible. This is a known limitation of the simulated environment.

---

**Q12: How does the wishlist handle duplicates?**

A: Two layers of protection. First, the `wishlist_items` table has a `UNIQUE` constraint on `(buyer_id, car_id)`. Second, in `WishlistServiceImpl.add()`, it uses `wishlistRepository.findByBuyerAndCar(user, car).orElseGet(...)` — if the item already exists, it's returned directly without trying to insert again. So the operation is idempotent.

---

**Q13: What is `@Transactional(readOnly = true)` and why use it?**

A: `readOnly = true` is a hint to the JPA provider and DB driver that no writes will occur in this transaction. Benefits: Hibernate skips dirty-checking (comparing entity states for changes), which improves performance. The DB driver may use a read-only connection or replica if available. It also serves as documentation — you know at a glance the method won't write anything.

---

**Q14: How are validation errors handled?**

A: DTOs use Jakarta Bean Validation annotations (`@NotBlank`, `@Email`, `@Pattern`, `@Min`, `@DecimalMin`, etc.). Controllers use `@Valid` on `@RequestBody` parameters. When validation fails, Spring throws `MethodArgumentNotValidException`. The `GlobalExceptionHandler` catches it, extracts the first field error, and returns a 400 with `{"message": "fieldName errorMessage"}`. Path variable validation uses `@Validated` at the controller class level and `@Positive` on parameters — failures throw `ConstraintViolationException`, also handled in the global handler.

---

**Q15: What does `spring.jpa.open-in-view=false` do?**

A: By default Spring Boot keeps a JPA `EntityManager` open for the entire HTTP request lifecycle (Open Session in View pattern). This allows lazy-loaded associations to be fetched during view rendering, but it ties DB connections to HTTP threads, hurting scalability. Setting it to `false` means all DB interaction must happen within `@Transactional` service methods. This is the correct production setting.

---

**Q16: How is the application configured for different environments (IDE vs production)?**

A: All sensitive/environment-specific values are injected via environment variables using Spring's `${VAR:default}` syntax in `application.properties`. Different `.env.*` files (`env.intellij`, `.env.eclipse`) configure the IDE to inject the right values as run-time environment. These files are git-ignored. An `.env.example` documents what variables are needed. For deployment (e.g., Vercel), the same env vars are set in the platform's dashboard.

## 22. NEW — Exception Handling Updates

### `MissingRequestHeaderException` handler

Before this fix, calling any protected endpoint **without the `X-Session-Token` header** returned 500. Now it returns a proper 401:

```java
@ExceptionHandler(MissingRequestHeaderException.class)
public ResponseEntity<ApiMessage> handleMissingHeader(MissingRequestHeaderException ex) {
    if ("X-Session-Token".equals(ex.getHeaderName())) {
        return ResponseEntity.status(HttpStatus.UNAUTHORIZED)
                .body(new ApiMessage("Login required"));
    }
    return ResponseEntity.badRequest().body(new ApiMessage(ex.getMessage()));
}
```

### `SupportTicket.responses` — `FetchType.EAGER`

The `responses` list on `SupportTicket` was `@OneToMany` with default lazy loading. With `spring.jpa.open-in-view=false`, the Hibernate session closes before Jackson serializes the response, causing:

```
LazyInitializationException: failed to lazily initialize a collection of role:
SupportTicket.responses — could not initialize proxy - no Session
```

Fix: `fetch = FetchType.EAGER` added to the annotation:

```java
@OneToMany(mappedBy = "ticket", cascade = CascadeType.ALL,
           orphanRemoval = true, fetch = FetchType.EAGER)
private List<TicketResponse> responses = new ArrayList<>();
```

**Why EAGER is acceptable here:** Support tickets always need their responses displayed together — it makes no sense to load a ticket without its messages. The collection is bounded (not a large unbounded list in production), so the N+1 risk is acceptable.

### Complete HTTP status code map

| Status | Trigger |
|---|---|
| `200` | Success |
| `400` | `ApiException(BAD_REQUEST)` — business rule violation |
| `401` | Missing header (`MissingRequestHeaderException`) or invalid/expired session |
| `403` | Valid session, wrong role or not the owner |
| `404` | `ApiException(NOT_FOUND)` |
| `409` | `ApiException(CONFLICT)` — duplicate username/email/phone/admin |
| `500` | Uncaught `Exception` — logged and generic message returned |

### Password minimum length

`RegisterRequest.password` and `AdminUpdateUserRequest.newPassword` both now carry `@Size(min = 10)`. This is enforced by Bean Validation before the service layer is even called. Error response:

```json
{ "message": "password must be at least 10 characters" }
```

## 23. NEW — Additional Interview Q&A

---

**Q17: What new condition fields were added to the Car entity and why?**

A: Nine fields: `condition` (EXCELLENT/GOOD/FAIR/POOR), `fuelType`, `transmission`, `bodyType` (free text), `numberOfOwners`, `engineCc`, `insured`, `pucValid`, and `description`. The motivation is to make the `/api/cars/compare` endpoint genuinely useful — the frontend can now render a full side-by-side comparison table without making any additional API calls. All fields are optional with sensible defaults so the change is backward compatible with existing clients.

---

**Q18: Why does `POST /api/cars` now return `CarListingResponse` instead of just `Car`?**

A: The `CarListingResponse` record wraps the `Car` with a `message` field: "Your car listing has been submitted successfully and is pending admin approval." This removes the need for the frontend to inspect `approvalStatus` to figure out what happened — the message tells the story directly. It also makes the API more self-documenting. `updateCar` still returns plain `Car` because updates don't have the same "you need approval" narrative.

---

**Q19: Explain the seller review upsert pattern.**

A: `SellerReviewServiceImpl.submitReview()` uses:
```java
SellerReview review = reviewRepository
    .findByReviewerAndSeller(reviewer, seller)
    .orElse(new SellerReview());
```
If a review exists for that reviewer-seller pair, it's fetched and updated in place. If not, a new one is created. Both paths call `reviewRepository.save()`. The `UNIQUE` constraint on `(reviewer_id, seller_id)` provides a DB-level safety net if two concurrent requests somehow slip through.

---

**Q20: How does the receipt number get generated and why isn't it stored?**

A: The receipt number is assembled at query time: `"RCP-" + order.getId() + "-" + order.getCreatedAt().toLocalDate()`. It doesn't need to be stored because it's fully deterministic — the same order always produces the same receipt number. Storing it would be redundant data. `generatedAt` is the only field that differs between calls and it's set to `LocalDateTime.now()` — it represents when the receipt was printed, not when the order was created.

---

**Q21: Why does admin password reset force a logout on the target user?**

A: After `AdminUserServiceImpl.updateUser()` hashes the new password, it nulls out `activeSessionTokenHash` and `activeSessionExpiresAt` on the target user:
```java
user.setActiveSessionTokenHash(null);
user.setActiveSessionExpiresAt(null);
```
This invalidates any active session the user has. They must re-authenticate with the new password. Without this, a user whose password was reset by admin could continue using their old session indefinitely until it naturally expired (up to 120 minutes).

---

**Q22: How does the 30-day graph data work and why are gaps pre-filled?**

A: `DashboardServiceImpl.adminGraphData()` first creates a `LinkedHashMap` with all 30 dates initialized to 0. It then fetches orders/cars from the last 30 days and increments the corresponding date. The result is always a complete 30-entry series with no holes. This simplifies frontend charting libraries — they can directly map the array to X-axis labels without gap detection. `LinkedHashMap` is used specifically to preserve insertion order (chronological).

---

**Q23: What was the `LazyInitializationException` bug and how was it fixed?**

A: `SupportTicket.responses` is a `@OneToMany` collection. By default JPA lazy-loads it — it's only fetched when accessed within an active Hibernate session. The app has `spring.jpa.open-in-view=false`, so the session closes after the service method returns. When Jackson tried to serialize the `responses` list in the controller response, the session was already gone → `LazyInitializationException` → 500 error. Fix: `fetch = FetchType.EAGER` forces Hibernate to load the responses in the same query (JOIN) when the ticket is fetched, so they're available when Jackson serializes.

---

**Q24: What does `MissingRequestHeaderException` represent and how is it handled?**

A: It's thrown by Spring MVC when a `@RequestHeader("X-Session-Token")` annotation is on a controller method but the client doesn't send that header at all. Previously this fell through to the generic `Exception` handler and returned 500. A dedicated `@ExceptionHandler(MissingRequestHeaderException.class)` was added to `GlobalExceptionHandler`. It checks the header name — if it's `X-Session-Token` it returns `401 {"message":"Login required"}`. Any other missing header returns 400 with the default Spring message.
